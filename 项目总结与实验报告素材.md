# CloudHome 个人主页项目总结与实验报告素材

## 一、项目概述

本项目是一个部署在云服务器上的个人主页系统，项目名称可概括为 `CloudHome Personal Site`。从当前代码、部署脚本、数据库脚本、Nginx 配置、Docker 编排文件、GitHub Actions 工作流以及服务器状态记录来看，该项目已经不是单纯的静态网页展示，而是一个具备“前端展示 + 后端接口 + 数据存储 + 云端部署 + 对象存储 + 缓存加速 + 自动化运维 + AI 对话”的综合性 Web 系统。

项目的核心目标是围绕“个人主页展示”这一主题，完成一个具有实际访问能力、可持续维护能力和云端运行能力的完整网站。它既承担了个人信息展示、经历介绍、技能展示、生活记录等内容呈现功能，也通过管理后台、留言系统、点赞系统、资源上传、备份恢复和安全巡检等能力，体现了完整的云计算课程实验特征。

从系统结构上看，项目采用了典型的前后端分离思路：

- 前端：使用 HTML、CSS、JavaScript 编写多个页面，负责内容展示、交互逻辑和后台管理界面。
- 后端：使用 FastAPI 提供 REST 风格接口，负责数据读写、权限校验、AI 转发、对象存储上传签名、邮件通知、运维调度等。
- 数据层：使用 SQLite 存储站内业务数据，使用 Redis 提供缓存和轻量级锁。
- 云端运行层：通过 Docker / Docker Compose 部署多个后端实例，通过 Nginx 做反向代理和负载均衡。
- 云服务层：通过阿里云 ECS 提供基础计算资源，通过阿里云 OSS 存储上传的图片与 PDF 文档。
- 运维自动化层：通过 Shell 脚本、systemd 定时器、GitHub Actions 实现自动部署、巡检、备份、自愈和恢复。

因此，这个项目本质上是一个“小型云原生风格个人网站系统”，它很好地把云计算课程中涉及的服务器部署、容器化、反向代理、对象存储、缓存、自动化运维、数据持久化和接口服务等内容串联在了一起。

## 二、项目当前已经实现的功能

### 1. 前台展示功能

根据 `frontend` 目录中的页面文件和相关脚本，项目目前已经实现了较完整的个人主页前台展示体系，主要包括：

#### （1）首页展示

首页 `index.html` 已实现以下内容：

- 个人主页欢迎页展示
- 个人身份介绍、专业方向、兴趣关键词等内容展示
- 个人形象图片展示
- 个人基础信息展示
- 技能标签展示
- 经历时间线展示
- 生活兴趣模块展示
- 首页内嵌 AI 对话入口
- 访客评价/点赞模块
- 留言墙模块

首页不是完全写死的静态页面。通过 `frontend/js/site-dynamic.js` 可以看出，首页部分文案会通过 `/api/content/get` 接口动态加载，例如：

- 首页标题
- 首页副标题
- 首页简介
- 留言墙简介

这说明项目已经实现了“静态页面骨架 + 后端可编辑内容”的混合设计。

#### （2）关于我页面

`about.html` 用于展示个人简介、学习经历、成长背景、兴趣倾向等内容，是个人主页的核心信息页面之一。

#### （3）技能展示页面

`skills.html` 对个人技能进行展示，并结合后端的 `/api/skills/documents` 接口动态加载技能相关 PDF 文档。也就是说，该页面不仅展示技能关键词，还支持展示外部技能证明材料或项目文档。

#### （4）经历展示页面

`experience.html` 展示个人学习经历、比赛经历、活动经历、志愿服务经历等内容。根据后端接口和前端动态脚本可知，经历页支持：

- 从后端读取经历列表
- 显示每条经历的标题、时间、描述
- 显示标签信息
- 为每条经历绑定多张图片

这使得经历模块从普通静态介绍，升级成了一个可以在后台持续维护的动态相册式经历系统。

#### （5）生活展示页面

`life.html` 主要用于呈现生活记录、旅行、阅读、摄影、音乐等内容，使网站不只是课程作业，也具有完整的个人展示属性。

#### （6）媒体/作品展示页面

`media.html` 结合图片和文档资源展示，属于站内资源页面的延伸模块。

#### （7）AI 助手页面

`ai.html` 是项目的重要特色页面之一，目前已经实现：

- 独立的 AI 对话界面
- 对话消息气泡渲染
- 快捷提问按钮
- 访客留言展示和提交
- Markdown 风格 AI 回复渲染
- 调用 `/api/ai-chat` 后端接口获得真实模型回复

从代码看，AI 页面保留过早期关键词匹配式问答逻辑，但当前页面已经接入真实后端 AI 接口，说明该模块经历了从“本地规则式对话”到“云端模型式对话”的演进。

#### （8）留言墙功能

前台留言功能已经实现：

- 访客填写昵称
- 留言正文输入
- 可选填写邮箱
- 可选择“收到回复时邮件通知”
- 提交留言到后端
- 展示所有留言
- 若管理员已回复，则同步展示回复内容

这说明项目具有真实的双向交互能力，而不是只读网站。

#### （9）点赞/评价功能

首页实现了访客对站主特征标签的点赞评价功能，后端维护点赞数量，前端本地通过 `localStorage` 避免重复点赞，形成了一套简易但完整的互动统计功能。

### 2. 后端接口功能

后端主程序位于 `backend/app.py`，已经实现了非常丰富的 API 能力。根据接口定义，目前主要功能如下。

#### （1）基础服务与健康检查

- `/api/health`：健康检查接口
- `/api/instance`：返回当前实例名
- `/api/cache/status`：查看 Redis 缓存状态
- `/api/cloud/status`：输出整个云部署架构和运行状态摘要

其中 `/api/cloud/status` 能直接返回当前系统涉及的：

- Nginx
- 负载均衡
- 双后端实例
- Docker Compose
- Redis
- SQLite
- 阿里云 OSS

这说明项目已经把“系统自描述”和“运行状态可观测性”纳入设计中。

#### （2）管理员认证功能

后台实现了基础管理员登录与鉴权机制：

- `/api/admin/login`
- `/api/admin/check`

认证方式为基于密码生成和校验 Bearer Token。虽然属于轻量级实现，但对于课程项目已经足够支撑后台维护。

#### （3）内容块动态维护功能

后端支持对页面文字内容进行后台维护：

- `/api/content/get`
- `/api/admin/content/update`

它的作用是把首页标题、副标题、介绍文案等抽取成数据库内容块，便于通过管理后台在线修改，而不必每次都手工改 HTML 文件。

#### （4）经历管理功能

后端已经实现完整的经历管理：

- `/api/experiences`
- `/api/admin/experience/add`
- `/api/admin/experience/delete`
- `/api/admin/experience/upload-image`

它支持：

- 新增经历
- 删除经历
- 为经历绑定图片
- 为经历设置标题、时间、正文、标签

这使网站具备轻量 CMS（内容管理系统）特征。

#### （5）技能文档管理功能

项目已经实现技能 PDF 文档管理：

- `/api/skills/documents`
- `/api/admin/skill/upload-pdf`
- `/api/admin/skill/delete`

支持管理员上传技能证明材料、比赛证明、实验报告等 PDF 文件，并在前台页面中展示。

#### （6）媒体资源管理功能

项目实现了站内媒体资源管理：

- `/api/admin/media/config-check`
- `/api/admin/media/init-upload`
- `/api/admin/media/complete-upload`
- `/api/admin/media`
- `/api/admin/media/list`
- `/api/admin/media/delete/{media_id}`

其完整流程是：

1. 后台发起上传初始化请求。
2. 后端校验文件类型、大小、重复上传指纹。
3. 后端通过 OSS SDK 生成带时效性的上传签名 URL。
4. 前端/后台将文件直接上传到对象存储。
5. 上传完成后再通知后端记录数据库元数据。

这是一种比较标准的“后端签名、前端直传对象存储”方案，明显优于把所有文件先传到应用服务器再二次上传。

#### （7）留言系统功能

留言系统后端接口已经完整实现：

- `/api/messages`
- `/api/messages/{message_id}/reply`
- `/api/messages/reply`
- `/api/messages/delete`

已支持：

- 游客提交留言
- 管理员查看留言
- 管理员回复留言

#### （8）邮件通知功能

项目实现了留言相关邮件通知能力：

- 新留言可通知管理员邮箱
- 管理员回复留言后，可通知访客邮箱

邮件配置使用 QQ 邮箱 SMTP，说明项目不仅做了页面交互，还考虑了实际通知链路。

#### （9）点赞统计功能

项目提供：

可用于读取和增加点赞计数，实现前台互动统计。

#### （10）审计日志功能

项目提供：

后端会对管理员登录、媒体上传、运维任务提交、通知事件等行为写入审计日志。这一点非常重要，因为它说明系统已经具备基本的可追溯性。

#### （11）后台仪表盘功能

项目提供：

该接口可以汇总：

#### （12）AI 助手后端功能

`backend/ai_module.py` 实现：

- `/api/ai-chat`

这一设计的最大意义在于：前端无需暴露 API Key，模型调用统一由后端代理完成，符合云端应用的基本安全原则。

它们共同构成了一个“可视化运维管理中心”，支持：

- 查看当前运维任务状态

## 四、项目所使用的云计算技术

这一部分是实验报告中最需要重点展开的内容。根据当前项目实现，已经实际使用了以下云计算相关技术。

### 1. 云服务器 ECS

项目部署在阿里云 ECS 上。代码中的系统提示词、脚本路径、部署结构和 `server-info` 目录里的状态记录都能证明这一点。

ECS 在本项目中的作用是：

- 提供公网访问能力
- 提供 Linux 运行环境
- 承载 Nginx、Docker、后端服务、数据库文件和运维脚本
- 作为整个个人主页系统的核心计算节点

这体现了 IaaS（基础设施即服务）的典型使用方式。

### 2. Nginx 反向代理与静态资源服务

项目使用 Nginx 作为 Web 入口，配置文件位于：

- `nginx/nginx.conf`
- `nginx/sites-available/default`

已实现的作用包括：

- 对外监听 80 端口
- 提供前端静态页面访问
- 将 `/api/` 请求反向代理给后端服务
- 通过 upstream 转发到多个 API 实例

这是云端 Web 应用中极为典型的入口层设计。

### 3. 负载均衡（应用层轮询转发）

Nginx 中定义了：

- `personal_api_cluster`
- `127.0.0.1:8001`
- `127.0.0.1:8002`

说明系统已经通过 Nginx upstream 将请求分发到两个后端实例，实现了基础的负载均衡能力。虽然不是云厂商专用 SLB，但已经体现了分布式服务部署和流量分摊思想。

### 4. Docker 容器化部署

后端使用 Docker 镜像封装运行环境，相关文件包括：

- `backend/Dockerfile`
- `backend/docker-compose.yml`

Docker 技术在本项目中的作用：

- 封装 Python 运行环境
- 固化依赖安装方式
- 保证部署环境一致性
- 方便复制多个后端实例

这属于课程中“应用容器化部署”的直接体现。

### 5. Docker Compose 多容器编排

`docker-compose.yml` 中已经定义多个服务：

- `redis`
- `api1`
- `api2`

说明项目已经具备：

- 多容器协同运行
- 服务间网络互联
- 统一生命周期管理
- 环境变量集中注入
- 数据目录与脚本目录挂载

这属于云应用部署中的“小型服务编排”能力。

### 6. Redis 缓存与分布式锁

项目引入 Redis，并在代码中实际用于两类任务：

#### （1）留言缓存

`/api/messages` 在普通访客访问时会优先查 Redis 缓存，缓存键为：

- `cloudhome:messages`

作用是：

- 减少 SQLite 读取次数
- 提升热点接口响应速度
- 提升前台并发访问性能

#### （2）上传锁

项目使用 Redis 的 `set nx ex` 机制实现上传去重锁，用于防止同一文件短时间内重复上传。

这说明 Redis 在本项目中不是“摆设”，而是已经真正参与业务流程。

### 7. 对象存储 OSS

项目通过 `oss2` SDK 接入阿里云 OSS。主要用途包括：

- 上传图片
- 上传 PDF
- 生成文件访问 URL
- 后台媒体资源管理
- 技能文档展示
- 经历图片展示

项目使用的是“签名上传 URL”方案，即：

- 后端鉴权
- 后端生成临时签名 URL
- 客户端直传 OSS
- 上传完成后回写数据库

这是一种非常规范的云存储使用方式，也明显体现了对象存储与应用服务器解耦的思想。

### 8. GitHub Actions 持续集成/持续部署（CI/CD）

`.github/workflows/deploy.yml` 表明项目已经建立自动化部署流水线，功能包括：

- 提交到 `main` 分支时自动触发
- 检查是否误提交 `.env`、数据库等敏感文件
- 检查超大文件
- 校验 Docker Compose 配置
- 编译检查 Python 文件
- 通过 SSH 连接云服务器
- 通过 SCP 同步前端、后端和脚本
- 在服务器上执行部署脚本
- 部署完成后验证 Redis、容器、运维状态

这说明项目已经具备“代码提交即自动部署”的 DevOps 特征。

### 9. 自动化运维脚本

项目在 `scripts` 目录中实现了多种运维脚本：

- `deploy.sh`
- `ops_runner.sh`
- `security_audit.sh`
- `backup_to_oss.sh`
- `restore_from_backup.sh`
- `self_heal.sh`

这些脚本体现了云计算系统运维中的多个关键能力：

- 自动部署
- 安全巡检
- 数据备份
- 数据恢复
- 故障自愈
- 定时任务执行

### 10. systemd 定时器 / 后台任务调度

部署脚本会在服务器上安装：

- `cloudhome-ops-runner.service`
- `cloudhome-ops-runner.timer`

定时器每 30 秒执行一次 `ops_runner.sh`，它负责轮询“待执行运维任务请求文件”，从而执行：

- 安全巡检
- 备份
- 恢复
- 自愈

这种设计本质上是一种轻量级异步任务调度机制，虽然没有引入 Celery、RabbitMQ 等复杂中间件，但非常适合课程项目，思路清晰且易维护。

### 11. 云端备份与恢复机制

项目的备份脚本会打包以下内容：

- SQLite 数据库
- `.env` 配置
- 安全报告
- 脚本目录
- Nginx 配置

并支持：

- 生成本地压缩备份
- 可选上传到 OSS
- 记录最新备份状态
- 列出历史备份
- 从指定备份恢复数据库和报告
- 恢复前自动创建 `pre-restore` 保护备份

这说明项目已经考虑了云端系统的容灾与回滚问题。

### 12. 自愈机制

`self_heal.sh` 会执行：

- 检查 nginx 服务
- 校验 nginx 配置
- 检查 docker 服务
- 检查核心容器是否运行
- 缺失或停止时重建容器
- 校验 API 健康状态
- 校验 Redis 连接状态
- 校验报告卷挂载状态

这体现了云服务运维中的“自动修复”思想，是一个很有实验亮点的部分。

### 13. 安全巡检机制

`security_audit.sh` 会检查：

- Nginx 配置是否正确
- Docker 容器是否运行
- Redis 是否暴露公网
- 后端端口是否暴露公网
- 首页是否可访问
- `/api/cloud/status` 是否可访问
- `/api/cache/status` 是否可访问
- CPU、内存、磁盘使用率
- 最近备份是否存在
- SSH 失败登录次数
- 防火墙状态

最终生成：

- 文本报告
- JSON 报告

这说明项目已经初步实现“可观测 + 可审计 + 可巡检”的运维体系。

## 六、项目运行流程说明

### 1. 普通页面访问流程

1. 用户通过公网访问 ECS 上的网站。
2. Nginx 接收 HTTP 请求。
3. 若请求是 HTML、CSS、JS、图片等静态资源，则由 Nginx 直接返回。
4. 若请求路径以 `/api/` 开头，则转发给后端 API 集群。
5. API 集群中的 `api1` 或 `api2` 实例处理请求并返回结果。

### 2. 动态内容加载流程

1. 前端页面加载完成后执行 JavaScript。
2. 前端调用 `/api/content/get`、`/api/experiences`、`/api/skills/documents` 等接口。
3. 后端从 SQLite 读取数据。
4. 前端将返回数据渲染到页面中。

### 3. 留言提交流程

1. 访客填写昵称、内容、邮箱等信息。
2. 前端调用 `/api/messages`。
3. 后端校验数据合法性。
4. 后端写入 SQLite。
5. 后端清除留言缓存。
6. 如果启用邮件通知，则给管理员发送新留言提醒。
7. 前端刷新留言列表。

### 4. 管理员回复留言流程

1. 管理员登录后台获取 Token。
2. 在后台查看留言列表。
3. 提交回复到 `/api/messages/reply`。
4. 后端更新 SQLite 中的 reply 字段。
5. 后端根据访客设置决定是否发送回复通知邮件。
6. 前台访客再次访问时可以看到回复结果。

### 5. 文件上传流程

1. 管理员选择图片或 PDF。
2. 前端向 `/api/admin/media/init-upload` 请求上传签名。
3. 后端校验文件类型、文件大小、重复上传风险。
4. 后端生成 OSS 临时上传 URL。
5. 前端将文件直接上传到 OSS。
6. 上传完成后调用 `/api/admin/media/complete-upload`。
7. 后端写入 `media_assets`。
8. 若是经历图片或技能 PDF，则再调用对应绑定接口。

### 6. AI 对话流程

1. 用户在 `ai.html` 或首页 AI 模块输入问题。
2. 前端调用 `/api/ai-chat`。
3. 后端读取系统提示词、模型配置和 API Key。
4. 后端请求大模型接口。
5. 后端解析结果后返回给前端。
6. 前端以 Markdown 风格渲染对话内容。

### 7. 自动部署流程

1. 开发者将代码推送到 `main` 分支。
2. GitHub Actions 自动执行检查任务。
3. 若检查通过，工作流通过 SSH/SCP 连接 ECS。
4. 同步前端、后端和脚本到服务器。
5. 执行 `deploy.sh`。
6. 脚本同步资源、备份旧配置、重建镜像、启动容器、安装定时器、重载 Nginx。
7. 工作流进一步验证 Redis、容器和运维状态。

## 九、当前项目仍存在的局限与可改进方向

为了写实验报告更客观，也建议写出项目当前的不足。根据现有代码，可以总结如下：

### 1. 数据库仍采用 SQLite

SQLite 对个人项目足够，但若访问量进一步上升，或者需要更强并发能力，后续可升级为 MySQL / PostgreSQL。

### 2. 管理员认证机制较轻量

当前是简化版 Token 机制，适合作业项目，但正式生产环境可进一步增加：

- 更严格的会话管理
- 多用户权限控制
- 登录失败限制
- HTTPS 和 CSRF 防护

### 3. 前端页面中仍存在历史演进痕迹

例如 AI 页面中保留过早期规则问答逻辑和多段脚本覆盖关系，说明代码经历了多次迭代。后续可以进一步重构，提高可维护性。

### 4. 运维任务队列为轻量文件队列

当前通过请求文件 + 定时轮询执行任务，思路很适合实验项目；若未来系统变大，可升级为消息队列或专业任务队列系统。

### 5. 缺少更系统化的监控体系

目前已有自定义状态接口和巡检脚本，但还未接入 Prometheus、Grafana 等专业监控工具。
